# 可选模块：环境适配设计

**适用条件**：skill 需要同时支持 Chat 和 Cowork，或对运行环境有特定要求时读取本文件。
纯对话类 skill（无文件操作、无脚本）天然双端兼容，不需要读本文件。

---

## Chat vs Cowork 核心能力差异

| 能力 | Chat | Cowork |
|---|---|---|
| 文件读写 | ❌（需 Desktop Commander MCP） | ✅ 直接用 Read/Write/Edit |
| 脚本执行 | ❌ | ✅ Bash 工具 |
| 状态文件存储 | ❌ | ✅ /tmp/ 可写 |
| 子任务并行 | ❌ | ✅ Agent 工具 |
| 系统 prompt 体积 | 较小 | 较大（工具定义多） |
| 对话历史跨 session | ✅ 可保留 | ❌ 每次新开 |

---

## 三种适配策略

**策略一：单环境设计**（推荐大多数情况）
明确主要运行环境，按该环境能力设计，不强求兼容。
选择依据：skill 是否有文件操作或脚本执行需求？有 → Cowork；无 → Chat/两者皆可。

**策略二：环境检测 + 分支**（需跨端支持时）
在 SKILL.md 路由层或对应工作流步骤里加：
```
环境检测：
- 有 Bash 工具 → 用 scripts/ 执行操作
- 无 Bash，有 Desktop Commander MCP → 用 MCP 执行操作
- 两者都无 → 输出操作步骤让用户手动执行
```

**策略三：最小公分母**（纯对话类 skill）
不依赖任何文件系统或脚本，响应即结果，无需适配。

---

## 工具定义的固定开销

Cowork 系统 prompt 包含所有工具定义（Bash、MCP 等），在第一条消息就全量载入，无法渐进控制。安装的 MCP 越多，基础 context 越大。这是平台层面的固定成本，skill 设计无法优化，只需知晓即可。

设计含义：在 Cowork 里，skill 自身的 SKILL.md 只是总 context 的一部分，更要保持精简。
