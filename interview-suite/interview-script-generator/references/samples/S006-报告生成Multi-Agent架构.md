# 问题：这个报告生成项目能讲讲吗？（通用版）

**场景**：被动回答 / 主动展示均适用
**话术功能**：传递信息 + 建立印象
**能力意图**：架构判断力、技术方案选型、工程可维护性思维
**讲述结构**：目标定义 → 问题本质重新定义 → 方案判断（单Agent的问题）→ 具体架构 → 架构价值 → 模型选型

---

## 话术

第二个功能是设计提案生成。设计师在项目收尾要花大量时间做报告，这部分时间应该还给创意本身。

报告生成表面上是一个生成任务，本质上是三件完全不同的事——检索资料、分析策略、可视化数据。

我考虑过单Agent，搭建成本更低，但这个任务链很长：单Agent跑到后面上下文压力大；三种性质完全不同的任务堆在一个prompt里，指令越来越长，LLM注意力分散，任务之间互相干扰，生成质量很难保证；出了问题不知道卡在哪里，中间过程不可拆分，难以定位，也不方便做评测。

具体架构是：主Agent负责拆解全局任务，任务规划Agent对每个节点做详细拆解，再交给执行Agent跑。为了让各Agent之间状态同步，加了一个全局上下文，让主Agent随时知道执行到哪一步、卡在哪里。

这样做的结果是：结构清晰，每个Agent职责边界明确，新加功能或改一个阶段不会影响其他Agent；可观测，每个阶段输入输出独立，出了问题能直接定位到哪个节点；好维护，哪个阶段质量不好单独优化那个Agent的prompt就行，不会牵一发动全身。这个取舍在长期来看是值得的。

模型选型上：主Agent和任务规划Agent用R1，负责全局拆解和规划；检索、分析、可视化三个执行Agent用V3，结构化输出。
