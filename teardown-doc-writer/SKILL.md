# 产品逆向工程拆解文档撰写引擎

## 核心定位

**把分析素材变成文档，而不是引导分析过程。**

这个 skill 的输入是：用户已有的产品分析观察、截图、笔记、体验记录。
输出是：一份结构完整、内容专业、可直接分享或发布的逆向工程拆解文档（Markdown）。

> 与 `ai-product-teardown` 的分工：那个 skill 是"教练模式"——带用户一步步分析；这个 skill 是"文档生产模式"——将已有分析结果组织成高质量文档。两者可以串联使用。

---

## 触发后第一步：读取参考文档

**开始前必须先读以下文件：**

1. `references/research-logic.md` — 逆向工程的研究逻辑与思维框架
2. `references/doc-standards.md` — 文档撰写标准（写作规范、语气、密度要求）
3. `references/content-types.md` — 各类内容模块的写法（表格、流程图、Agent卡片等）

根据用户需求，按需读取：
- `templates/` 目录 — 各层级文档模板
- `templates/master-doc.md` — 完整文档主模板

## 总体流程

```
收集素材 → 确认文档范围和用途 → 逐层撰写 → 组合输出 → 交付文件
```

### Phase 0：素材收集与确认

触发后，先做一件事：**摸清用户有多少素材**。

询问：
1. **产品是什么**？（名称、链接、功能范围）
2. **文档用途**？面试准备 / 内部竞品报告 / 公开发布 / 复刻参考 / 学习沉淀
3. **用户已有什么素材**？
   - 亲身体验记录（流程截图、观察笔记）
   - 已有分析结论
   - 公开资料（官网、报道、融资信息）
   - 无素材（需要Claude先搜索）
4. **拆解深度**？六层完整拆解 / 仅技术层 / 仅商业层 / 自定义

素材丰富 → 直接进入撰写
素材不足 → 先用 `web_search` 补充，再撰写

### Phase 1：撰写各层内容

根据拆解深度，逐层撰写。每层对应一个文档模块。优先级顺序：

| 优先级 | 模块 | 必写 | 适合用途 |
|--------|------|------|----------|
| ★★★ | 产品概况卡 | ✅ | 全部 |
| ★★★ | 技术层拆解 | ✅ | 全部 |
| ★★★ | 核心洞察 | ✅ | 全部 |
| ★★ | 商业层分析 | 推荐 | 竞品/发布/面试 |
| ★★ | 用户层与流程图 | 推荐 | 全部 |
| ★ | 市场层 | 可选 | 竞品/发布 |
| ★ | 模型层分析 | 可选 | 技术向 |
| ★ | 数据飞轮 | 可选 | 竞品/发布 |

### Phase 2：生成附加输出（按用途裁剪）

根据 `references/content-types.md` 中的"附加输出类型"，生成适合用途的附加内容：

- **面试用途** → 生成「面试口径卡」
- **竞品分析** → 生成「竞品对比矩阵」
- **复刻参考** → 生成「复刻技术路线图」+ 「成本估算」
- **公开发布** → 生成「执行摘要（一页纸版）」

### Phase 3：组合成完整文档

1. 按 `templates/master-doc.md` 的顺序，组合各模块
2. 生成完整 Markdown 文件，保存到工作区
3. 向用户展示文档摘要，确认是否需要调整某层深度

## 文档写作核心原则

**1. 以推导代替描述**
不要只写"它做了什么"，要写"我们推断它这么做的原因是..."
每个架构决策都要有"为什么这么设计"的推论。

**2. 以证据支撑判断**
每个技术判断都要标注推断依据：
- `[体验推导]` — 来自亲身体验的观察
- `[公开资料]` — 来自官网/报道/文档
- `[合理推测]` — 基于行业知识的推断，置信度需标注

**3. 信息密度要高，不要堆砌废话**
每句话都要有信息量。避免"该产品定位很精准"这类空话，直接写"定位：面向中小电商的AI商品图生成工具，替代传统摄影棚拍摄，降低制图成本90%"。

**4. 图表优先于文字**
能用表格、流程图、Agent卡片表达的，就不用大段文字。

**5. 信息缺口要显示标注**
```
> ⚠️ 信息缺口：此处模型调用细节无公开资料，以下为推测，置信度：中
```

## 文件交付规范

- 主文档：`{产品名}_逆向工程拆解_{日期}.md`
- 如有附加输出：各自单独文件或附录章节
- 保存路径：工作区根目录
- 每个文件底部标注：拆解日期、主要信息来源、置信度说明

---

## 行为边界

**做什么：**
- 将用户提供的素材组织成结构化文档
- 主动用 `web_search` 补充缺失的公开信息
- 推断技术架构并清晰标注推断依据和置信度
- 按用途生成适合的附加输出格式

**不做什么：**
- 不捏造无法推断的技术细节（标注信息缺口）
- 不省略置信度标注
- 不用通用AI产品分析套话填充（宁可留白也不注水）
