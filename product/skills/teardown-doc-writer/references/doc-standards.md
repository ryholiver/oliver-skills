# 文档撰写标准

## 文档整体结构

一份完整的逆向工程拆解文档包含以下模块（按照在文档中出现的顺序）：

```
1. 文档封面与元信息
2. 执行摘要（可选，适合对外分享版）
3. 产品概况卡
4. 市场层分析（可选）
5. 商业层分析
6. 用户层分析
7. 技术层拆解 ← 核心
   7.1 架构类型判断
   7.2 全局上下文 Schema
   7.3 Agent 架构图
   7.4 Agent 详细拆解（每个Agent一张卡片）
   7.5 产品问题诊断（可选）
8. 模型层分析（可选）
9. 基础层/数据飞轮（可选）
10. 核心洞察与结论
11. 附录（按用途选择）
    A. 评测打分表
    B. 竞品对比矩阵
    C. 复刻技术路线图
    D. 面试口径卡
```

---

## 各层写作规范

### 产品概况卡

格式：信息密集的卡片式。用表格或结构化列表，一屏内看完核心信息。

必须包含：
- 产品名称 + 链接
- 一句话定位（谁/场景/替代了什么）
- 核心功能（3条以内）
- 当前状态（融资轮次/用户规模/主要市场）
- 拆解范围与深度
- 文档版本与日期

示例写法：
```markdown
| 字段 | 内容 |
|------|------|
| 产品 | OiiOii（https://oiioii.com） |
| 定位 | 面向中小电商的AI商品图生成工具，替代传统摄影棚拍摄 |
| 核心功能 | 商品图生成 / 模特换装 / 场景替换 |
| 当前状态 | 2024年B轮，主要市场：跨境电商 |
| 拆解范围 | 商品图生成完整流程（技术层重点） |
| 拆解日期 | 2026-03-26 |
```

---

### 市场层

适用：竞品分析、对外发布、深度研究场景。面试准备可跳过或仅写一段话。

标准格式：
1. **赛道判断**（一段话：规模/增速/阶段/关键趋势）
2. **竞争格局表**（产品/定位/融资/差异化策略）
3. **市场窗口判断**（这个产品找到了什么缝隙？）

写作要求：
- 数据有来源（即使是估算也要注明来源和估算逻辑）
- 竞品表格要写"差异化策略"列，不能只列名字
- 不要写"市场前景广阔"这类空话，直接写数字

---

### 商业层

标准格式：
1. **一句话定位**（严格格式：为[谁]在[场景]下解决[问题]，替代了[什么旧方案]）
2. **商业模式**（变现方式 + 定价策略 + 成本结构估算）
3. **护城河分析**（技术/数据/网络效应/品牌/工艺知识）
4. **产品气质**（创始人价值观、审美、战略方向——从访谈中提炼）

写作要求：
- 护城河必须评级（高/中/低）并给出判断依据
- "产品气质"不能跳过——创始人的背景和理念决定了很多非理性但关键的产品决策

---

### 用户层

标准格式：
1. **正向流程图**（用户完成核心任务的完整步骤）
2. **逆向流程图**（出错、不满意、中途退出时的处理路径）
3. **Human-in-the-Loop节点清单**（哪些步骤需要用户做决策）
4. **评测打分表**（见附录A模板）
5. **关键体验观察**（5-8条最有信息量的发现，每条有分析，不只是描述）

流程图格式（ASCII示例）：
```
用户输入商品图
    ↓
[系统] 背景检测 + 主体分割
    ↓
系统推荐3个场景模板 ← [用户决策点1] 选择或自定义
    ↓
[系统] 场景生成（~15秒）
    ↓
展示4个候选结果 ← [用户决策点2] 选择/重新生成/微调
    ↓
[用户] 下载/继续编辑
```

写作要求：
- 关键体验观察每条必须写"观察→分析"结构，不能只描述
  - ❌ "系统会在生成前展示进度条"
  - ✅ "生成过程展示了带文字的进度条（'正在分析商品特征'→'正在生成场景'），这暗示背后可能不是单次API调用，而是多个串行步骤 [体验推导]"

### 技术层（最重要）

#### 7.1 架构类型判断

格式：
```markdown
**架构类型**：Multi-Agent 协作型
**四象限定位**：Workflow确定性中 × Context确定性低

**判断依据**：
- [体验推导] 相同输入会产生不同的中间步骤路径，说明有规划能力
- [体验推导] 观察到至少3个不同"身份"的处理阶段
- [公开资料] 官网技术博客提到"多智能体协同"

**置信度**：高
```

#### 7.2 全局上下文 Schema

格式：YAML代码块，带注释说明每个字段的用途和被哪些Agent读写。

```yaml
# 全局上下文 Schema（推导版，置信度：中）
# 注：此Schema从各Agent的输入输出反推而来，非官方文档

project_context:
  # 项目元数据
  project_id: "string"        # 会话唯一ID
  created_at: "timestamp"
  status: "analyzing|generating|reviewing|done"  # 状态机

  # 输入资产
  input_assets:
    original_image: "url"     # 用户上传的原始图片
    product_category: "string" # 商品品类 [分析Agent写入]
    product_features: []      # 提取的产品特征 [分析Agent写入]

  # 用户偏好（在各步骤中累积）
  user_preferences:
    selected_style: "string"  # 用户选择的风格
    approved_scenes: []       # 已通过的场景

  # 生成结果
  outputs:
    candidates: []            # 候选图片列表
    final_selection: "url"    # 用户最终选择
```

#### 7.3 Agent 架构图

格式：ASCII树形图，标注数据流方向

```
输入处理层
├── ImageAnalysisAgent      ← 主体分割 + 商品识别
└── ContextBuildAgent       ← 构建全局上下文

规划层
└── ScenePlannerAgent       ← 根据商品类型规划场景方案

生成层（并行）
├── SceneGeneratorAgent × 4 ← 并行生成4个候选场景
└── QualityCheckAgent       ← 过滤不合格结果

交互层
└── ReviewCoordinator       ← 呈现结果 + 处理用户反馈
```

#### 7.4 Agent 卡片

每个核心Agent使用标准卡片格式（见 `templates/agent-card.md`）。

写作要求：
- 工具集要写**推断依据**，不能凭空捏造
- "规则边界"这栏最有价值，要认真填写——这往往暴露产品的安全机制和质量控制
- Prompt骨架仅在"复刻"用途下输出，其他情况可选

#### 7.5 产品问题诊断（可选）

格式：
```markdown
### 问题：[描述现象]
- **现象**：[观察到的具体表现]
- **可能根因**：Prompt问题 / 工具调用设计 / 流程设计 / 模型能力
- **影响程度**：高/中/低
- **如果我是PM的修法**：[具体建议]
```

---

### 模型层

标准格式：
1. **模型路由表**（功能模块 / 推测模型 / 推断依据 / 可替换性）
2. **模型依赖度评估**（哪些环节强依赖模型，哪些靠工程实现）
3. **成本结构推算**（可选，基于公开API定价估算）

---

### 基础层（数据飞轮）

标准格式：
1. **数据资产清单**（类型 / 来源 / 用途 / 可复制性 / 竞争优势）
2. **飞轮分析**：用户增长是否反哺数据 → 数据是否改善产品 → 闭环是否成立
3. **护城河评级**（与商业层的护城河分析相互印证）

---

### 核心洞察与结论

这是全文最重要的部分。格式：

```markdown
## 核心洞察

### 洞察1：[一句话概括]
[2-3句话解释为什么这个洞察重要，支撑证据是什么]

### 洞察2：...

## 针对不同用途的结论

### 竞品分析视角
[如果你是竞品PM，最需要关注的3件事]

### 复刻参考视角
[最难复制的部分是什么？最容易复制的是什么？]

### 学习研究视角
[这个产品在哪些设计决策上值得借鉴？]
```

## 写作风格规范

### 语气与语调

- **专业但不学术**：写给产品经理/工程师/创业者看，不写给评审委员会看
- **直接**：先结论后理由，不绕弯子
- **有立场**：拆解报告要有判断，不是中立的功能列表

### 密度要求

好句子 vs 坏句子：

| ❌ 低密度（删掉）| ✅ 高密度（保留）|
|---|---|
| "该产品的用户体验较为流畅" | "主流程从上传到出图<30秒，零等待感，但批量处理(>10张)时会出现队列等待，无进度反馈" |
| "技术架构较为复杂" | "Multi-Agent，至少4个Agent串行，推测总token消耗约2000-4000/次 [合理推测]" |
| "市场竞争激烈" | "同赛道已有Photoroom(B2C)、Remove.bg(API)、美图(国内)，但聚焦跨境电商商品图的PMF目前OiiOii最清晰" |

### Markdown格式规范

- 每个层级标题用 `##` 或 `###`，不用 `####`（层级不超过3层）
- 表格用于对比性信息（竞品/评测/数据资产）
- 代码块用于Schema/提示词/架构图
- 引用块（`>`）用于信息缺口标注
- 粗体（`**`）只用于真正需要强调的词，不要整句加粗

## 附加输出类型

### A. 执行摘要（一页纸版）

适合：对外分享、展示给领导/客户看
长度：500字以内
内容：产品是什么 → 最重要的3个发现 → 核心结论一句话

### B. 面试口径卡

适合：准备AI PM面试
格式：Q&A形式，5-8个高频问题 + 参考回答框架
内容：产品是什么？技术架构怎么推断的？亮点和不足各是什么？如果你来做PM会怎么改？

### C. 竞品对比矩阵

适合：竞品分析报告
格式：矩阵表格（产品 × 维度）
维度建议：定位/核心功能/技术深度/定价/护城河/适合场景

### D. 复刻技术路线图

适合：创业参考/学习
内容：MVP最简实现路径 / 技术难点排序 / 成本估算 / 可复用的开源组件

### E. 提示词骨架（可选，仅限深度拆解）

适合：深度技术学习/复刻
内容：关键Agent的提示词骨架（角色定义/步骤/工具调用规范/约束），标注"推导版"
