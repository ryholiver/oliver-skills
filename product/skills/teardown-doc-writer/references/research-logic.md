# 逆向工程研究逻辑

## 核心思想：从外到内，从现象到机制

逆向工程产品拆解的本质，是**用已知的外部现象（用户界面、交互行为、输出结果），反推内部的设计决策和技术实现**。

研究路径的方向感：
```
市场背景（为什么存在）
    ↓
商业设计（赚什么钱、服务谁）
    ↓
用户体验（用户实际看到和感受到什么）
    ↓
技术架构（背后是怎么跑的）
    ↓
模型选择（哪些环节依赖AI、用什么模型）
    ↓
数据资产（护城河在哪、飞轮是否存在）
```

---

## 六层研究逻辑详解

### Layer 1：市场层 — 赛道值不值得进入？

**研究问题：**
- 这个赛道的市场规模和增速如何？
- 主要竞争对手是谁？各自的定位差异？
- 技术成熟度处于哪个阶段（Gartner曲线）？
- 是否有明显的市场窗口期？

**推导逻辑：**
市场层是"为什么这个产品存在"的背景。拆解不是为了写行业报告，而是为了理解**产品在竞争格局中的位置**——它找到了什么尚未被满足的缝隙？还是在已有市场上做差异化？

**信息获取路径：**
- 搜索行业报告关键词：`{赛道} 市场规模 2024/2025`
- 搜索融资信息：`{产品名} 融资轮次`
- 搜索竞品关系：`{赛道} 竞争格局 主要玩家`

**文档中的呈现形式：**
- 竞争格局一览表（产品/定位/融资/差异点）
- 赛道阶段判断（一段话+判断依据）

---

### Layer 2：商业层 — 这是什么生意？

**研究问题：**
- 它为谁解决了什么问题（一句话定位）？
- 定价策略和变现模式是什么？
- 护城河来自哪里（技术/数据/网络效应/品牌）？
- 创始人的战略意图是什么？

**推导逻辑：**
商业层回答"这个生意能不能活"。核心是找到**产品的价值主张和变现逻辑之间的匹配关系**。

关键追问：
- 如果底层模型能力变强到"100分"，这个产品的价值还剩什么？→ 这暴露了真正的护城河
- 它的用户愿意付费的核心原因是什么？→ 功能？效率？体验？信任？

**信息获取路径：**
- 抓取官网首页 + 定价页
- 搜索创始人访谈：`{创始人} 产品理念 访谈`
- 搜索产品发布报道

**文档中的呈现形式：**
- 一句话定位（谁+场景+替代了什么）
- 商业模式拆解表（收入来源/定价/成本结构）
- 护城河评级（高/中/低 + 判断理由）

---

### Layer 3：用户层 — 用户真正完成的任务是什么？

**研究问题：**
- 用户的完整使用旅程是什么（正向流程）？
- 哪些节点需要用户做决策（Human-in-the-Loop）？
- 出错或结果不满意时，用户能怎么处理（逆向流程）？
- 产品的核心体验亮点和痛点分别是什么？

**推导逻辑：**
用户层是技术层拆解的基础——你只有先搞清楚用户的完整操作路径，才能推断背后有哪些Agent在工作、各自的触发时机是什么。

流程分析的关键技巧：
- **每个步骤都问**：这里系统做了什么决策？用户做了什么选择？
- **找状态切换点**：界面的每次切换背后往往对应一次API调用或Agent切换
- **记录异常处理**：故意给模糊输入，观察系统如何处理不确定性

**文档中的呈现形式：**
- 正向流程图（ASCII或Mermaid）
- 逆向流程图（异常/回退路径）
- 评测打分表（按维度打分 + 扣分理由）
- 关键体验观察（列举5-8条最有信息量的观察）

---

### Layer 4：技术层 — 它是怎么实现的？（核心层）

**研究问题：**
- 整体架构类型：固定Workflow / 单Agent / Multi-Agent / 混合？
- 有哪些Agent？各自的职责边界是什么？
- Agent之间的调用关系：链式 / 主从 / 并行 / 混合？
- 全局上下文Schema是什么？哪些字段贯穿全程？
- 每个Agent的输入-处理-输出是什么？
- 它调用了哪些外部工具（搜索/代码执行/数据库/渲染）？

**推导逻辑：**

**第一步：判断架构类型**

用四象限框架判断：

|  | Workflow确定性高 | Workflow确定性低 |
|--|--|--|
| **Context确定性高** | 固定Workflow/RPA型 | 工作流+AI节点型 |
| **Context确定性低** | 单Agent自主规划型 | Multi-Agent复杂协作型 |

判断依据：
- 相同输入→相同流程：偏向Workflow
- 展示思考/规划过程：偏向Agent
- 多角色/多身份轮流工作：Multi-Agent

**第二步：识别Agent边界**

把产品想象成一个人类团队——需要哪些岗位？每个岗位的职责是什么？然后将岗位映射为Agent。

Agent边界识别技巧：
- 界面有明显的阶段切换（如"分析中→生成中→审校中"）
- 输出的格式/风格有明显变化（可能换了Agent）
- 等待时间有明显差异（可能是不同计算量的Agent）

**第三步：逐个拆解每个Agent**

对每个Agent，推导：
```
输入：接收什么数据/指令
处理：执行什么逻辑（Plan-and-Execute vs ReAct）
工具集：调用了哪些工具
输出：产出什么结果
规则边界：有什么不能做/必须遵守的约束
```

**第四步：推导全局上下文Schema**

把所有Agent的输入输出整合——找出**被多个Agent共同读写的字段**，这就是全局上下文。

Schema应包含：
- 项目元数据（ID、状态、版本）
- 核心业务字段（与产品功能直接相关）
- 资产字典（角色/场景/素材/模板等）
- 历史记录/会话上下文

**注意顺序**：Schema是最后推导出来的，但在文档中放最前面展示。

**文档中的呈现形式：**
- 架构类型判断（+ 判断依据）
- 全局上下文Schema（YAML格式）
- Agent架构图（ASCII树形图）
- 每个Agent的Agent卡片（见模板）
- 产品问题诊断（如有明显问题）

---

### Layer 5：模型层 — 哪些环节依赖什么模型？

**研究问题：**
- 哪些环节明确调用了AI模型？
- 能否判断用的是什么模型（通过速度/风格/响应特征推断）？
- 是否根据任务复杂度动态切换模型？
- 哪些环节其实不依赖强模型，靠工程实现？

**推导逻辑：**
模型层的价值在于区分"模型依赖型价值"和"工程设计型价值"。
- 如果模型达到100分，产品还剩什么？→ 工程价值
- 如果工程实现完美，但模型不行，产品还能用吗？→ 模型依赖度

**文档中的呈现形式：**
- 模型路由表（功能模块/推测模型/推断依据/是否可替换）
- 模型依赖度评估（高/中/低 + 理由）

---

### Layer 6：基础层 — 数据飞轮是否存在？

**研究问题：**
- 产品运转需要哪些数据资产？
- 这些数据是用户生产的、还是平台预置的？
- 用户规模增长是否能反哺数据质量？（飞轮是否成立）
- 这是护城河还是负担？

**文档中的呈现形式：**
- 数据资产清单（类型/来源/用途/可复制性）
- 飞轮分析（是否存在 + 当前处于飞轮的哪个阶段）

## 研究过程的注意事项

### 关于推断置信度

所有技术层以下的内容都是推断，不是事实。必须区分：

| 标注 | 含义 | 使用场景 |
|------|------|----------|
| `[体验推导]` | 来自亲身使用观察 | 流程/Agent行为判断 |
| `[公开资料]` | 来自官网/报道/文档 | 商业/市场信息 |
| `[合理推测]` | 基于行业知识的推断 | 技术架构/模型选型 |
| `[信息缺口]` | 无法确认的内容 | 标注未知项 |

置信度等级：高/中/低，在文档结尾汇总标注。

### 关于信息缺口

信息缺口不是文档的失败，而是文档的诚实。宁可写"此处无法确认"，也不用套话填充。

标注方式：
```
> ⚠️ 信息缺口：[描述不知道的具体内容]。推测方向：[如果非得猜，猜什么]。建议验证方式：[如何能获取这个信息]。
```
