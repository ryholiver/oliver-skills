# 知识地图格式规范

> 本文件在 Step 2 生成知识地图前读取，其余步骤无需加载。

---

## Markdown 知识地图格式

```markdown
## 📍 知识地图 · [文档标题] · [日期]

---

### 🗂 主题组一：[组名，反映这组话题的核心矛盾或问题]
*[一句话说明：这组话题在解决什么问题，为什么值得关注]*

---

**知识点 A：[标题]** ⬜ 未开始

**背景**：[这个话题为什么出现，触发它的具体情境，前因后果]

**核心结论**：
- 🔑 [可直接借用的实践细节]
- ⚠️ [需要验证的说法] 🔗 [链接或搜索词]
- 📌 [背景信息，了解即可]

**你可能会问**：
- [预置问题1——帮助用户发现自己不知道的盲区]
- [预置问题2]
- [预置问题3]
```

**进度状态标记**（初次生成时全部为 ⬜）：
- `⬜ 未开始`
- `🔄 进行中 · [卡在什么地方]`
- `✅ 已吸收 · [日期]`

---

## 标注规则

### 🔑 关键实践
满足全部三条：
1. 需要真实做过才能知道（不是搜索能直接得到的）
2. 有具体操作或结论，不是泛泛而谈
3. 对想了解该领域的人有直接参考价值

### ⚠️ 需要验证
满足任意一条即标记：
- 说话人用了不确定语气（"好像""我感觉""可能"）
- 具体数字无来源（"效率提升30%""支持100万token"）
- 我的知识库与之有冲突信息
- 涉及具体项目名/版本号无法确认

### 📌 背景铺垫
- 帮助理解其他知识点的前置信息
- 去掉不影响核心知识理解
- 受众已知的常识不标注

---

## ⚠️ 项附链接规则

按以下顺序处理，遇到能解决的就停：

1. **自身知识库有把握** → 直接给出链接
2. **知识库不确定** → 触发 WebSearch，附最相关的官方或 GitHub 链接
3. **仍无法找到** → 给出精准搜索词，格式：`` `site:github.com OpenManus agent` ``

不要给出无法确认真实性的链接。

---

## HTML 知识地图所需 JSON 格式

生成 HTML 前，将知识地图内容转换为以下 JSON 结构：

```json
{
  "title": "<文档标题>",
  "date": "<YYYY-MM-DD>",
  "groups": [
    {
      "name": "主题组名称",
      "summary": "一句话说明这组话题的核心矛盾或问题",
      "cards": [
        {
          "id": "card-唯一标识",
          "title": "知识点标题",
          "status": "todo",
          "background": "背景描述——这个话题为什么出现，前因后果",
          "conclusions": [
            {"type": "key",     "text": "可直接借用的实践细节（不含emoji前缀，脚本会自动加）"},
            {"type": "warning", "text": "需要验证的说法", "link": "https://...或搜索词"},
            {"type": "note",    "text": "背景信息"}
          ],
          "questions": [
            "预置问题1",
            "预置问题2",
            "预置问题3"
          ]
        }
      ]
    }
  ]
}
```

**card id 命名规则**：
- 用英文或拼音，全小写，用连字符分隔
- 要能望文生义（如 `card-langgraph-state`、`card-agent-loop`）
- 同一文档内不重复
- 后续 `update_progress.py` 用这个 id 定位卡片

## 进度恢复看板格式

检测到已有知识地图文件时，展示：

```
📊 这份文档你之前处理过

✅ 已吸收：知识点A · 知识点B
🔄 进行中：知识点C · 上次卡在：工具调用循环那里
⬜ 未开始：知识点D · 知识点E · 知识点F

从哪里继续？
```

如果有对应的 HTML 文件，提醒用户在浏览器中打开查看完整地图。
