# Skill Extractor

从任意一个已跑通的工作流线程中，提取构建 Skill 所需的六个核心要素。

## 输入

用户提供一条已经成功执行过的工作流线程（可以是 Codex 对话、文档、Slack 记录、Runbook 等），从中提取可复用的流程。

## 执行工作流

### Step 1: 提取六要素
依据"六个提取要素"框架，从线程中提取：触发条件、输入材料、执行步骤、验收标准、可复用资源、后续更新方式。

依据框架：

```markdown
#### 六个提取要素

对每条线程，提取以下六个维度：

##### 1. 触发条件（When to Trigger）
- 什么情况下应该使用这个 Skill？
- 用户会用哪些关键词或短语调用它？
- 它解决了什么类型的重复任务？

##### 2. 输入材料（Inputs）
- 使用这个 Skill 时，用户需要提供什么？
- 文件、链接、上下文、样例、模板
- 哪些是必需的，哪些是可选的？

##### 3. 执行步骤（Steps）
- 从这条成功线程中，哪些步骤应该保留？
- 顺序是什么？
- 有没有分支逻辑或条件判断？

##### 4. 验收标准（Acceptance Criteria）
- 什么结果算合格？
- 哪些错误必须避免？
- 有没有质量红线？

##### 5. 可复用资源（References / Scripts / Assets）
- 哪些提示词、命令、模板、样例可以放进 references？
- 有没有可执行的脚本（scripts/）？
- 有没有静态资源（assets/）？

##### 6. 后续更新方式（Maintenance）
- 未来发现遗漏时，如何把修正写回 Skill？
- 多久 review 一次？
- 谁负责更新？
```

### Step 2: 输出草案供确认
按"输出格式"模板输出六要素草案，提交用户确认。如有要素信息不足，标注「待确认」。

输出格式：

```markdown
## Skill 设计草案

### 触发条件
...

### 输入材料
...

### 执行步骤
...

### 验收标准
...

### 可复用资源
...

### 后续更新方式
...
```

### Step 3: 调用 skill-creator 生成 SKILL.md
用户确认草案后，使用 `skill-creator` 生成正式的 SKILL.md，不要直接写文件。

## 工作原则

- **从真实任务提炼，不要凭空设计**。如果线程信息不足以填满某个要素，明确标注「待确认」。
- **保持通用性**，避免把 Skill 写得太窄——它应该适用于同一类型的所有任务，而不只是这一条具体对话。
