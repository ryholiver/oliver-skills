# Skill 渐进式披露审查清单

## 五个审查维度

### 维度 1：体积
**目标**：SKILL.md 正文 ≤ 150 行（不含 frontmatter）

| 行数 | 评级 | 说明 |
|------|------|------|
| ≤ 100 行 | ✅ 优秀 | 无需优化 |
| 101–150 行 | ⚠️ 尚可 | 注意不要继续增长 |
| 151–250 行 | 🔴 偏大 | 建议压缩 |
| > 250 行 | 🔴🔴 严重 | 必须重构 |

**如何计算**：去掉 frontmatter（`---` 之间的内容）和注释后，统计实际行数。

---

### 维度 2：条件分支密度
**识别方法**：查找 `if...else`、`当用户说...`、`模式A/B/C`、`触发...时` 等结构。

每个分支如果有 **>20 行**的专属内容，就是迁移候选。

**评分**：
- 0–1 个大分支：✅
- 2–3 个大分支：⚠️ 建议每个分支独立 references 文件
- 4+ 个大分支：🔴 必须拆分

**典型案例（obsidian-knowledge-saver）**：
- 目录管理模式A（新增目录）：约 30 行 → 应迁移到 `references/mode-directory-add.md`
- 目录管理模式B（调整顺序）：约 25 行 → 应迁移到 `references/mode-directory-reorder.md`
- 目录管理模式C（同步）：约 30 行 → 应迁移到 `references/mode-directory-sync.md`
- Vault 初始化流程：约 70 行 → 应迁移到 `references/init.md`

---

### 维度 3：模板/示例密度
**识别方法**：查找大段代码块（``` ``` ```）、输出模板、示例文件内容。

单个代码块/模板 **>15 行** 就是迁移候选。

**评分**：
- 无或极少：✅
- 1–2 个大模板：⚠️ 可接受，或迁移到 `references/templates.md`
- 3+ 个大模板：🔴 必须迁移，放到 `assets/` 或 `references/templates.md`

---

### 维度 4：初始化/配置流程
**识别方法**：查找"首次使用"、"初始化"、"检测操作系统"、"读取配置文件"等步骤。

**原则**：初始化逻辑通常只在第一次运行时触发，不应占用每次调用的 context。

**处理方式**：
```
references/init.md  ← 所有初始化步骤

SKILL.md 中只留一行：
> 首次使用时，读取 `references/init.md` 完成配置初始化。
```

**评分**：
- 无初始化逻辑：✅
- 有初始化逻辑且已在 references/：✅
- 有初始化逻辑但在 SKILL.md 中 ≤ 10 行：⚠️ 可接受
- 有初始化逻辑且在 SKILL.md 中 >10 行：🔴 必须迁移

---

### 维度 5：可脚本化操作
**识别方法**：查找每次执行都需要的固定操作：文件写入、路径拼接、目录扫描、编号递增等。

**原则**：凡是"给定输入，确定输出"的操作，都可以写成脚本放在 `scripts/`，Claude 不需要重新思考逻辑，直接调用即可。

**可脚本化操作示例**：
- 写入 Markdown 文件到指定路径
- 按日期前缀递增编号（001, 002...）
- 扫描目录找相关文件名用于双向链接
- 读取 directory-map.md 并解析表格
- 路径拼接（Mac/Windows 兼容）

**评分**：
- 无重复性固定操作：✅
- 1–2 个可脚本化操作但逻辑简单：⚠️
- 有明显重复性固定操作（每次调用都要执行）：🔴 建议脚本化

---

### 维度 6：状态管理适配性
**识别方法**：看 skill 的工作流复杂度与它的状态保存机制是否匹配。

**需要状态管理的信号**：
- 工作流 > 4 步且有多个"等待用户确认"节点
- 预期任务会跨越打断或多次对话
- 有跨 skill 调用（参数型或上下文型）

**审查项**：
- 有状态管理需求但没有任何状态标注机制 → 🔴
- 有文字状态标注但格式冗长（段落式）→ ⚠️ 建议改为紧凑 `[STATE:...]` 格式
- 运行在 Cowork 但仍用文字标注而非文件存储 → ⚠️ 可优化
- 简单 skill（≤4步）但设计了复杂状态机制 → ⚠️ 过度设计，删掉

**评分**：
- 无状态需求，无状态机制：✅
- 有状态需求，有合适机制：✅
- 有状态需求，无任何机制：🔴
- 状态机制与复杂度不匹配（过简或过繁）：⚠️

---

### 维度 7：跨 Skill 调用设计
**识别方法**：查找 skill 中是否涉及"调用/触发另一个 skill"或"依赖外部工具执行子任务"。

**审查项**：
- 有跨 skill 依赖但没有写明 → 🔴（用户需要手动识别并切换，体验差）
- 写了跨 skill 调用但没有标注关系类型（参数/上下文/插曲/探索）→ ⚠️
- 没有写降级方案（子 skill 不可用时怎么办）→ ⚠️
- 把所有跨 skill 逻辑都堆在 SKILL.md 而不是 references/ 工作流文件 → ⚠️

**评分**：
- 无跨 skill 依赖：✅（不适用）
- 有依赖且设计完整（类型+交接+降级）：✅
- 有依赖但设计不完整：⚠️ 或 🔴（视缺失严重程度）

---

### 维度 8：执行模式声明
**识别方法**：查找 skill 是否在 SKILL.md 或工作流文件里声明了默认执行模式。

**适用范围**：有文件修改、不可逆操作、或多步骤需要用户确认的 skill 才需要检查。

**评分**：
- 有不可逆操作但无执行模式声明 → 🔴（用户不知道 Claude 会不会直接执行）
- 有声明且与任务复杂度匹配 → ✅
- 简单 skill（纯信息输出）无声明 → ✅（不适用，不扣分）
- 声明了但与任务复杂度不符（复杂任务用直接执行）→ ⚠️

---

### 维度 9：理由输出机制
**识别方法**：查找 skill 的执行流程里是否有自动解释每步原因的内容。

**评分**：
- 执行流程里有大量"这样做是因为..."的固定解释 → 🔴（强制输出，浪费 token）
- 无理由机制，但也没有"用户问才解释"的说明 → ⚠️（模糊，可能导致过度或过少解释）
- 有 `<!-- rationale: -->` 注释或明确说明"解释条件" → ✅
- 理由完全不存在且 skill 设计决策复杂 → ⚠️（维护困难）

---

## 综合评分与优先级

| 综合得分 | 建议 |
|--------|------|
| 全部 ✅ | 无需优化 |
| 1–2 个 ⚠️ | 低优先级，可选优化 |
| 任意 🔴 | 高优先级，建议重构 |
| 多个 🔴 | 紧急，严重影响 token 效率 |

---

## Token 影响估算

每次 skill 触发时，SKILL.md 全文都会载入 context。粗略估算：

- 1 行 ≈ 20–50 个 token（中文更多，代码块更少）
- 150 行 ≈ 3,000–5,000 tokens（可接受）
- 450 行 ≈ 9,000–15,000 tokens（obsidian-knowledge-saver 的现状，每次调用多耗 10k tokens）

**迁移收益估算**：
```
可迁移行数 × 30 tokens/行 ≈ 每次节省的 tokens
```

例如：obsidian-knowledge-saver 可迁移约 300 行 → 每次节省约 **9,000 tokens**
