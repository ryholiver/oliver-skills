# skill-optimizer 设计决策说明

本文件记录 skill-optimizer 自身的主要设计选择，供用户查阅。不自动输出，用户问才引用。

---

## 为什么 SKILL.md 只有路由逻辑，不放工作流细节？

每次 skill 触发，SKILL.md 全文都会载入 context。把工作流细节全放在里面，等于每次调用都付出完整内容的 token 成本，哪怕当前任务只用到其中一个模式。把细节推到 references/ 后，只有被触发的分支才进入 context。

---

## 为什么 design-workflow.md 要拆出 design-optional/ 目录？

design-workflow.md 本身是一个"设计工作流"，但并非每次设计 skill 都需要状态管理、跨 skill 调用、环境适配。如果把这些全放在主文件里，每次设计 skill 时都要读完整个文件，即使这次的 skill 根本不涉及跨 skill。拆成 optional/ 模块后，只有满足条件的 skill 才需要读对应文件。这是该 skill 对自身设计原则的实践。

---

## 为什么执行模式默认是"计划优先"而不是"直接执行"？

本 skill 的主要操作是修改 skill 文件，属于不可逆操作（修改后难以完整恢复原状）。
"计划优先"在执行前给用户一个确认机会，防止用户不知情的情况下文件被修改。
代价是多一轮交互，但对于文件修改类任务，这个代价是值得的。

---

## 为什么"对话意图识别"从 SKILL.md 删掉了？

本 skill 使用"计划优先"执行模式，用户确认后才执行操作。在这个模式下，即使 Claude 误读了用户意图，也只会在计划确认阶段暴露，用户可以纠正，不会造成实际损失。因此意图识别原则在这里是冗余的。该原则保留在 `design-optional/execution-mode.md` 里，作为"直接执行模式"的附注，仅在必要时适用。

---

## 为什么状态标注放在回复末尾而不是开头？

状态在步骤完成后才能确定，放在末尾是步骤完成时的即时记录，最准确。
放在开头则是预设状态，可能与实际执行结果不符。
此外，末尾标注是对话中最近的内容，Claude 往回扫描时最先找到。

---

## 为什么 save_state.py 用 /tmp/ 而不是 skill 目录？

skill 目录在 Cowork 中通常是只读的（`.claude/skills/` 路径受保护）。
`/tmp/` 是 Cowork 沙箱中可写的标准临时目录，适合存放 session 内的状态文件。
代价是 session 结束后 /tmp/ 会清空，但状态文件本来就是 session 级别的，不需要跨 session 持久化。
