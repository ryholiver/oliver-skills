# 模式 B：设计新 Skill 框架

## 第一步：捕捉核心意图

一次问清楚（不需要分多轮）：

1. 这个 skill 要完成什么核心任务？（一句话）
2. 有几种不同的执行路径/模式？
3. 有没有只在第一次运行时才需要的初始化步骤？
4. 每次调用都会执行的固定操作是什么？（文件写入、路径拼接等）
5. 主要运行环境：Chat / Cowork / 两者都要？

---

## 第二步：规划三层结构

根据第一步回答，画出目录树：

```
[skill-name]/
├── SKILL.md              ← 路由层：任务目标 + 模式路由 + 必要声明
├── references/
│   ├── [mode-a].md       ← 模式A完整工作流（步骤>5或>30行时单独抽出）
│   ├── [mode-b].md       ← 模式B完整工作流
│   └── init.md           ← 初始化流程（如有）
└── scripts/
    └── [operation].py    ← 固定操作脚本（如有）
```

判断标准：
- 工作流 ≤ 5 步且 < 30 行 → 可留在 SKILL.md
- 工作流 > 5 步或 > 30 行 → 单独 references/ 文件
- 每次都执行的固定操作（写文件、拼路径）→ scripts/

---

## 第三步：写 SKILL.md 路由层

SKILL.md 只写判断逻辑，不写执行细节：

```markdown
## 任务目标
[一段话]

## 模式路由
- 用户意图是 [A] → 读取 references/[mode-a].md
- 用户意图是 [B] → 读取 references/[mode-b].md
- 不确定时 → 问：[具体询问语]
```

---

## 第四步：填写 references/ 工作流文件

每个文件结构：步骤名 → 操作说明 → 完成标准。

写作原则：
- **确定性操作**（文件写入、路径拼接、格式转换）→ 规定具体，不留模糊空间
- **判断性操作**（内容提炼、分类归属、创意选择）→ 给约束条件，让 Claude 运用判断
- **用户能力扩展场景**（用户和 Claude 都不确定时）→ 写"在满足[条件]的前提下自主选择并说明依据"

**理由输出的设计决策**：直接调用 AskUserQuestion（不输出前置文字）：
- A：默认不解释，用户问才解释（推荐，节省 token）
- B：关键步骤自动输出理由（在对应步骤加 `<!-- rationale: 原因 -->` 注释）

---

## 第五步：判断需要哪些可选模块

逐条检查，满足条件则读取对应文件：

| 条件 | 读取 |
|---|---|
| 工作流 > 4 步 或 预期会被打断 | `design-optional/state-management.md` |
| 有步骤需要调用另一个 skill | `design-optional/cross-skill.md` |
| 需要同时支持 Chat 和 Cowork | `design-optional/environment-adapt.md` |
| 有不可逆操作或需用户确认 | `design-optional/execution-mode.md` |

每读完一个模块，将对应内容加入 SKILL.md 或 references/ 文件，再继续下一个。

---

## 第六步：验证清单

核心项（必查）：
- [ ] SKILL.md 路由逻辑不看 references/ 也能做决策？
- [ ] 每个 references/ 触发条件在 SKILL.md 里明确标注？
- [ ] 确定性操作已规定清楚，判断性操作保留了探索空间？
- [ ] 初始化逻辑是否单独隔离？

读过可选模块的额外检查（按需）：
- [ ] 状态管理格式紧凑可被快速定位？Cowork 用了文件存储？
- [ ] 跨 skill 调用标注了类型、阻塞性、降级方案？
- [ ] 环境适配策略是否明确？
- [ ] 执行模式是否已声明？
