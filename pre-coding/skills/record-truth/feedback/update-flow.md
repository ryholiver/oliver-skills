# 更新流程

满意后的修正结果触发——从修正过程中提炼规则，倒推优化事实提取流程，让下次使用不需要多轮修改。

## 触发条件

修正子流程结束后，用户说"可以了"，选择"要更新规则"时触发。

## 流程

```
修正子流程结束（满意）
       ↓
修正记录写入 correction-log.md
       ↓
Claude 分析修正记录，提炼偏好
       ↓
向用户展示提炼结果，询问是否更新
       ↓
  是 ──→ 更新 profile.md
       → 更新 SKILL.md 对应规则
       → 下次使用直接生效
  否 ──→ 记录保留，不更新规则
```

## 修正记录结构

每次修正子流程结束后，在 `correction-log.md` 追加一条记录：

| 字段 | 说明 |
|------|------|
| 涉及事实类型 | 哪类事实（架构决策/编码约定/配置规则/模块边界/技术绕路） |
| 原始问题 | 原本的判断有什么偏差 |
| 修正内容 | 最终改成了什么样 |
| 反映偏好 | 这次修正反映出你对事实判断的什么倾向 |

## 偏好画像

修正日志积累到一定量后（或你主动说"整理修正记录"），Claude 分析修正日志，提炼你的偏好模式，写入 `profile.md`。

画像示例：

```markdown
## 模板选择调整
- 「架构决策」模板：必须有明确的候选方案对比，单方案不记

## 排除规则
- 环境变量相关的决策优先归入「技术绕路」而非「配置规则」
```

## 更新 SKILL.md

偏好画像确认后，Claude 将新的规则写入 SKILL.md 的对应位置：
- 判断条件变化 → 更新步骤 2 中的原则或判断框架
- 模板匹配变化 → 更新步骤 3 中的模板匹配条件
- 新的判断规则 → 追加到对应部分

## 重要原则

- **是否更新由用户决定**：Claude 只提供分析结果，不自动修改规则
- **一条修正不推翻整个体系**：只有多个相同偏向的修正积累后，才值得提炼为规则
- **修正记录永远保留**：即使用户选择不更新规则，修正日志也不删除
