# 初学者学习代码

## 目的

这份规则的目标不是让 AI 写生产级代码，而是让 AI 写出适合我阅读、理解和模仿的第一版代码。

我当前的学习方式是:

- 先让 AI 按功能要求写出最小可读实现
- 我先阅读代码，理解代码是怎么工作的
- 等我理解之后，再考虑如何改造成更完整、更工程化的版本

所以默认情况下，AI 写代码时应优先考虑"容易读懂"，而不是"最佳实践最完整"。

## 总原则

- 只实现我明确要求的功能，不额外扩展
- 优先写"最小可读实现"，不是"最完整实现"
- 优先让我看懂，再考虑工程质量
- 优先直接、朴素、展开的写法，不要炫技
- 不要为了显得专业而主动增加复杂度

## 写代码时的默认要求

- **必须单文件实现**，除非拆文件后分别超过 200 行。不要为了"职责分离"或"看起来专业"而拆文件
- 一个需求只做一件事。用户要命令就只注册命令，不要顺手加工具、快捷键、自动补全等
- 优先少函数、少抽象、少层级。把逻辑直接写在需要的地方，不要提前抽函数
- 如果一个函数只会调用一次，优先把它内联，除非内联后明显更难读
- 优先直白的控制流，不要绕
- 变量名可以长一点，但要清楚
- 能直接写清楚的逻辑，就不要提前封装
- 能用最普通的写法解决，就不要用技巧性写法
- 如果你参考了别人的代码或文档，不要照搬它的结构。用你自己的理解写出最简版本

## 默认不要主动加的东西

除非我明确要求，或者功能本身离不开，否则默认不要主动加入以下内容:

- 参数校验
- 复杂错误处理
- 日志系统
- 配置系统
- 依赖注入
- 设计模式
- 过早抽象
- 泛化的工具函数
- 为未来扩展预留的大量接口
- 测试代码
- 性能优化
- 完整的边界处理
- 生产级安全处理
- 兼容多种运行时的额外分支
- 为了通用性增加的配置解析
- 流式输出、事件流解析、状态同步等"更完整"的交互逻辑
- 临时文件、中间层包装、辅助模块，除非当前功能离不开

**判断方法：** 写完代码后，逐行检查。如果删掉某一行代码，功能仍然能跑，那这一行就不该在第一版里。

## 遇到官方示例时

- 官方示例常常是"完整示例"，不是"教学第一版"
- 默认不要跟着官方示例把功能做全，只借用最核心的调用路径
- 动手前先问自己：这个示例里哪些东西是这次需求根本不需要的？先删掉这些，再开始写
- 如果官方示例里有多模式、多配置、多兼容分支，默认都不要抄，除非用户明确要

## 强制删减清单

涉及插件、扩展、agent、SDK、CLI 这类任务时，写完后必须再检查一遍，下面这些东西能删就删:

- frontmatter 解析
- 自定义配置格式
- model、tools、provider 之类的透传配置
- JSON 流解析
- 临时文件
- 兼容 node/bun/不同入口的分支
- 单独的工具函数文件
- 只用一次的小函数
- "以后可能会用到"的参数

## 关于框架、SDK 和 API 文档

如果任务涉及框架、SDK、第三方库或 API:

- 默认先查 Context7，再写代码
- 优先参考最新、准确、与当前版本匹配的文档
- 不要凭记忆补全 API
- 如果文档里有最小示例，优先采用最小示例的思路
- 写出来的代码要尽量贴近官方文档能看懂的风格

## 输出代码时的风格要求

- 代码以"教学第一版"为目标
- 不要把一个简单需求写成企业级架构
- 不要为了复用而过早拆分
- 不要为了优雅而降低可读性
- 如果一种写法更短但更难懂，不要用
- 如果一种写法更长一点但更清楚，优先用更清楚的

## 解释代码时的要求

写完代码后，应使用简洁中文解释:

- 这段代码先做了什么
- 接着做了什么
- 数据是怎么流动的
- 哪一部分是这次功能的核心

解释时不要讲太多空泛概念，尽量贴着代码本身讲。

## 如果必须增加复杂度

如果你判断某些复杂度确实不能省略，不要直接铺开写很多。应该:

- 先尽量保留最小实现
- 只加入当前功能运行所必需的部分
- 明确说明为什么这部分不能去掉

## 我当前最想学到的东西

我想先学会下面这些能力:

- 一个功能最小是怎么跑起来的
- 框架代码最核心的骨架是什么
- 一个 SDK 最基本的调用路径是什么
- 代码从输入到输出是怎么流动的
- 哪些部分是"必须的"，哪些部分只是"工程增强"

所以请始终围绕"帮助我看懂第一版代码"来写。

## 一句话原则

请默认给我写: 功能正确、结构最小、容易阅读、适合初学者理解的第一版代码。

*** 注意，一定注意:

1. 如果我没特意提到，能不增加的功能就全都不增加，保持最小可运行。
2. 代码都写完了，也要再审核一遍，这个代码还能再简化吗，应该去掉哪部分，可以让代码最简短。
3. 再问自己: 还能再简化吗？
4. 如果你已经写出一个能跑的版本，不要急着交付，先主动做一轮"暴力删减"，把兼容性、抽象、配置、通用性相关代码尽量砍掉。
5. 如果用户要的是第一版，不要默认帮他保留"以后扩展会方便"的结构。以后要扩展，到时候再加。
