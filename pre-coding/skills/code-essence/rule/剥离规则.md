# 剥离规则

本文件定义从生产代码剥离骨架时的具体规则。剥离原则参考 `beginner-learning-code` 的正向规则（写代码时"不要加什么"），反向使用——遇到以下内容，砍掉。

## 核心判断方法

> 如果删掉某一行代码，核心功能仍然能跑，那这一行就不该留在骨架里。

## 通用规则

- 默认单文件输出，除非核心逻辑天然跨文件
- 只保留"你当前想看的功能"，不保留同文件中其他功能的代码
- 多层抽象 → 内联成直线流程
- 只调用一次的函数 → 内联回调用处
- 原代码的短变量名 → 重命名为大白话
- 复杂/绕的写法 → 展开成直白顺序
- 只保留 happy path，异常/边界分支砍掉
- 除非核心逻辑依赖，否则砍掉回调/异步等待

## 强制删减清单

剥离完成后再检查一遍，以下内容能删就删：

- 参数校验、输入格式校验、权限校验
- 复杂错误处理、异常捕获（只保留能让流程走通的必要处理）
- 日志输出、日志级别判断
- 配置读取、环境变量解析、配置文件加载
- 依赖注入容器、工厂函数、策略模式、设计模式
- 接口定义、抽象类（除非核心逻辑直接依赖）
- 工具函数、helper 类（内联到使用处或砍掉）
- 性能优化代码（缓存、懒加载、并发控制、池化）
- 兼容性分支（node/bun/不同版本/不同平台的判断）
- 流式输出、事件流解析、状态同步等异步交互
- 临时文件、中间层包装、辅助模块
- 类型定义（如果删掉不影响理解核心逻辑）
- 通用的、预留的、"以后可能会用到"的参数或分支
- 测试代码
- 生产级安全处理
- 为了通用性增加的配置解析
- 代码注释（如果代码本身已经够清楚）
